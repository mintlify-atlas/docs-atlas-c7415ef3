---
title: "Parallel Changes"
description: "Work on multiple changes simultaneously and use bulk-archive to close them all at once with conflict resolution."
---

OpenSpec tracks each change in its own folder under `openspec/changes/`. This makes it straightforward to context-switch between work streams and pick up where you left off.

## Working on multiple changes at once

You can start a new change at any time, even if another change is in progress. The following example shows switching from an in-progress feature to an urgent bug fix:

```text
You: I'm working on add-dark-mode but got pulled into a bug fix

You: /opsx:new fix-login-redirect

AI:  Created openspec/changes/fix-login-redirect/

You: /opsx:ff

AI:  ✓ Created planning artifacts

You: /opsx:apply

AI:  Working on fix-login-redirect...
     [Implements the fix]
     All tasks complete!

You: /opsx:archive

AI:  ✓ Archived fix-login-redirect

You: Let's get back to dark mode

You: /opsx:apply add-dark-mode

AI:  Resuming add-dark-mode...
     Picking up at task 2.3: Update Header...
```

**Best for:** Parallel work streams, urgent interrupts, team collaboration.

## Resuming a change

To resume a specific change, pass its name as an argument to `/opsx:apply`:

```bash
/opsx:apply add-dark-mode
```

Completion state is tracked in `tasks.md` checkboxes. OpenSpec picks up from the first unchecked task.

## Naming changes clearly

Good change names make `openspec list` useful and make it easier to target specific changes with commands.

| Good | Avoid |
|------|-------|
| `add-dark-mode` | `feature-1` |
| `fix-login-redirect` | `update` |
| `optimize-product-query` | `changes` |
| `implement-2fa` | `wip` |

## Bulk archive

When you have multiple completed changes, use `/opsx:bulk-archive` to close them all at once. It detects spec conflicts across changes and resolves them by checking what's actually implemented in the codebase.

**Full conversation example:**

```text
You: /opsx:bulk-archive

AI:  Found 3 completed changes:
     - add-dark-mode (all tasks done)
     - fix-login-redirect (all tasks done)
     - update-footer (all tasks done)

     Checking for spec conflicts...
     ⚠ add-dark-mode and update-footer both touch specs/ui/

     Inspecting codebase to resolve...
     Both changes are implemented. Will apply in chronological order.

     Archive all 3 changes?

You: Yes

AI:  ✓ Archived add-dark-mode
     ✓ Archived fix-login-redirect
     ✓ Archived update-footer
     Specs merged in order: add-dark-mode → update-footer
```

When multiple changes touch the same specs, bulk-archive inspects the codebase to confirm both are implemented, then merges in chronological order.

## The archive process in detail

Whether you use `/opsx:archive` or `/opsx:bulk-archive`, each change goes through the same steps:

<Steps>
  <Step title="Check artifact status">
    OpenSpec confirms that all expected artifacts exist (`proposal.md`, `specs/`, `design.md`, `tasks.md`) and reports task completion.

    ```text
    Artifact status:
    ✓ proposal.md exists
    ✓ specs/ exists
    ✓ design.md exists
    ✓ tasks.md exists (12/12 tasks complete)
    ```

    Archive won't block on incomplete tasks, but it will warn you.
  </Step>
  <Step title="Sync delta specs">
    If the change's delta specs haven't been synced to the main `openspec/specs/` directory, archive prompts to sync now.

    ```text
    Delta specs: Not yet synced to main
    → Would you like to sync now?
    ```

    For bulk-archive, conflicts across changes are resolved first by inspecting the codebase, then specs are merged in chronological order.
  </Step>
  <Step title="Move to archive folder">
    The change folder moves to `openspec/changes/archive/YYYY-MM-DD-<name>/`. All artifacts are preserved for audit history.

    ```text
    ✓ Synced specs to openspec/specs/auth/spec.md
    ✓ Moved to openspec/changes/archive/2025-01-24-add-auth/

    Change archived successfully.
    ```
  </Step>
</Steps>

## Best practices for focused changes

One logical unit of work per change. If you find yourself doing "add feature X and also refactor Y", consider two separate changes.

**Why it matters:**

- Easier to review and understand
- Cleaner archive history
- Can ship independently
- Simpler rollback if needed

<Tip>
  Use `/opsx:verify` before archiving to catch mismatches between your implementation and artifacts. It won't block the archive, but it surfaces issues you might want to address first.
</Tip>

<Warning>
  Bulk-archive prompts before overwriting spec content when conflicts are detected. Review the proposed merge order before confirming.
</Warning>
